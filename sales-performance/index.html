Problem

Manual, biased, and slow

Sales bonus calculations at UPC Broadband were done manually every month — a process that took 5 to 6 working days and was prone to errors. Worse, the existing performance metrics allowed reps to cherry-pick the easiest territories, skewing results and making fair comparison impossible.

Management had no reliable way to compare rep performance across territories or identify where coverage gaps existed.

Solution

Flat-level assignment tracking

The system tracked performance at the individual flat (apartment unit) level — each rep was assigned specific flats, and conversions were measured against that exact territory. This eliminated the ability to cherry-pick easy areas and made performance metrics genuinely comparable.

SQL Server · rep_performance.sql
-- Rep performance by assigned territory
WITH assignments AS (
  SELECT
    rep_id,
    flat_id,
    assignment_date,
    COUNT(flat_id) OVER (
      PARTITION BY rep_id
    )               AS total_assigned_flats
  FROM territory_assignments
  WHERE month = @target_month
),

conversions AS (
  SELECT
    rep_id,
    flat_id,
    sale_date,
    product_type,
    contract_value
  FROM sales
  WHERE month = @target_month
)

SELECT
  a.rep_id,
  r.rep_name,
  r.region,
  a.total_assigned_flats,
  COUNT(c.flat_id)          AS total_conversions,
  SUM(c.contract_value)    AS total_revenue,

  -- Conversion rate against assigned territory
  ROUND(
    COUNT(c.flat_id) * 100.0
      / NULLIF(a.total_assigned_flats, 0),
    2
  )                          AS conversion_rate_pct,

  -- Bonus calculation
  CASE
    WHEN conversion_rate_pct >= 15 THEN SUM(c.contract_value) * 0.08
    WHEN conversion_rate_pct >= 10 THEN SUM(c.contract_value) * 0.05
    ELSE                              SUM(c.contract_value) * 0.03
  END AS bonus_amount

FROM assignments a
JOIN reps r          ON a.rep_id = r.rep_id
LEFT JOIN conversions c
  ON a.rep_id = c.rep_id
  AND a.flat_id = c.flat_id
GROUP BY
  a.rep_id, r.rep_name, r.region,
  a.total_assigned_flats
ORDER BY conversion_rate_pct DESC

BRIO frontend for management

The SQL layer was surfaced through a BRIO reporting frontend, giving managers a monthly performance dashboard. Territory coverage gaps were visible at a glance, and the bonus figures were auto-calculated — no manual spreadsheet work required.
